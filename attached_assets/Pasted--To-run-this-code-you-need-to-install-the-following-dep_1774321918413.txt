# Project Sentinel

**Autonomous, Persistent RPG World Engine.**

## Architecture Skeleton

Sentinel operates on a strict separation of concerns to enable seamless remote play via a Tailscale mesh network.

1. **Inference Node**: Houses the DM, Fact-Extractor, and Lorekeeper agents. It evaluates user input, queries the world state, and outputs rich narrative alongside machine-readable tags.
2. **Infrastructure Node**: The Linux-based storage layer. It manages the PostgreSQL/Vector database, background simulations, and a Git-backed hybrid filesystem (JSON for state, Markdown for lore).
3. **The MCP Bridge**: The Inference node *never* touches files directly. It issues structured requests to local MCP servers on the Infrastructure node, which validate and execute filesystem, database, and git operations.

## The Core Loop

1. **Action**: User inputs a role-play action via the Client Node.
2. **Narrative**: DM Agent generates the story response.
3. **Extraction**: Fact-Extractor parses the response for state changes.
4. **Trigger**: System generates a structured `<world_update>` JSON tag.
5. **Execution**: Relevant MCP server (Filesystem/DB) consumes the tag and updates the Infrastructure.
6. **Sync**: Git-Sync MCP commits the change to version control.

## Getting Started (Bridge Initialization)

To bring up the skeletal infrastructure and establish the MCP bridge:

```bash
# 1. Spin up the Postgres/Vector databases
cd infrastructure

## 4. The Sentinel Airlock (Anonymization Manifest)
To support the safe sharing of `.spak` world-states, the export pipeline enforces strict anonymization rules, ensuring compliance with 2026 GDPR and EU AI Act (Privacy by Design / Data Minimization) mandates.

### 4.1. The Redaction Taxonomy
Before any `/data` folder is bundled into a `.spak`, the pipeline actively scrubs:
- **PII:** Real user names, physical addresses, and player profile handles.
- **Network Identifiers:** Tailscale IPs, local MAC addresses, and hostnames.
- **Credentials:** Any `.env` fragments, API keys (e.g., OpenAI, Anthropic keys), and database passwords.

### 4.2. Tokenization Strategy (Narrative Continuity)
To prevent the redaction process from destroying the context of the Lorekeeper's RAG memory, scrubbed data is replaced with **Repeatable Tokens**:
- If a player named \"John Doe\" (playing the character \"Kael\") accidentally types their real name into a session log, the Airlock regex engine swaps it for `<REAL_NAME_1>`.
- Every subsequent mention of \"John Doe\" in that specific `.spak` generation uses `<REAL_NAME_1>`, preserving the semantic relationships in the Markdown without leaking identity.

### 4.3. Metadata Stripping
- **Images:** Any community-uploaded maps or NPC portraits have their EXIF data (GPS coordinates, camera models, timestamps) recursively purged using `exiftool` equivalents in the pipeline.
- **Git History:** The Git database (`.git/`) is **never** included in the `.spak`. Only the current state tree is exported, preventing a malicious actor from reverting to a previous commit that contained unredacted data.

### 4.4. The 'Human-in-the-Loop' Flag
The Anonymization Pipeline utilizes a hybrid Regex + NLP detection layer.
- **Trigger:** If the system detects a high-risk entity pattern that regex struggles to confidently classify (e.g., an unbroken string of numbers resembling a Credit Card, or a string formatted like a Social Security Number or street address).
- **Action:** The automated `.spak` build is immediately halted. The export is flagged `NEEDS_REVIEW`, and the Infrastructure Node notifies the user to manually inspect and redact the flagged line in the session log.

### 4.5. Compliance (GDPR/AI Act 2026)
This multi-layered approach guarantees **Data Minimization** (exporting only the state required to run the game, stripping local user configs) and **Privacy by Design** (making anonymization the default, un-bypassable state of the export tool).
